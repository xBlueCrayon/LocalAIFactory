# LAF Enterprise ERP

A small but real, end-to-end **double-entry ERP** built on **.NET 10 / ASP.NET Core** with
**EF Core** over **SQL Server or SQLite**. It implements a clean-room version of the patterns common
to open-source ERPs — a draft/submit/cancel document lifecycle, a maker/checker approval workflow, an
immutable general ledger, and a moving-average stock ledger — and proves them with an automated test
suite.

This product was generated by the **LocalAIFactory** platform. It is a complete, buildable solution,
not a mock-up or a stub.

---

## Clean-room / trademark note

LAF Enterprise ERP is an **independent, clean-room implementation**. It is inspired by the *concepts*
that ERP systems such as ERPNext/Frappe popularised (a `docstatus` 0/1/2 document lifecycle, naming
series, a role-permission matrix, chart-of-accounts roots), but it **contains no ERPNext, Frappe, or
other third-party application code**, copies no schema, and uses **none of their trademarks**. All
entity names, table names, and logic here are original. Where a concept is borrowed, the source code
documents it as "clean-room equivalent of …" so the lineage of the *idea* is explicit while the
*implementation* is wholly this project's own.

"Frappe" and "ERPNext" are trademarks of their respective owners and are referenced here only to
describe prior art; this project is not affiliated with, endorsed by, or derived from them.

## What it is (honest scope)

**Implemented and tested:**

- **Accounting** — double-entry GL posting for sales invoices, purchase invoices, payments and
  journal entries; the engine refuses to persist an unbalanced voucher; cancellation posts reversing
  entries rather than editing history.
- **Stock** — an immutable signed stock ledger with **moving-average valuation**, a **negative-stock
  guard**, and voucher reversal on cancel.
- **Workflow (maker/checker)** — submit / approve / reject with maker≠checker enforcement, an
  amount threshold (under-threshold auto-approves, over-threshold needs a separate approver),
  mandatory rejection reasons, and an audit event on every transition.
- **RBAC** — a role-permission matrix (read/create/write/submit/approve/cancel per doctype).
- **Reports** — General Ledger, Trial Balance, Stock Balance, and AR/AP totals.
- **CRM / Projects / Support / Assets** — light operational modules (leads→customer conversion,
  project tasks with approval, support ticket escalation/resolution, asset maintenance scheduling).
- **CSV import/export** of customers, with per-row error capture.
- **REST API** (minimal APIs) and **9 read-oriented Razor pages** plus a dev-auth login.

**Deliberately NOT implemented (be aware before using this as a production ERP):**

- No real **authentication** — identity is selected from a dev cookie on the login page. There is no
  password/credential storage, no encryption, and no field-level permissions.
- **UI is read-only** — the Razor pages are views plus the dev-auth login form. There are **no
  create/edit forms** in the UI; records are created through the REST API or the service layer.
- No **Profit & Loss, Balance Sheet, aging, or period closing** reports.
- No **Manufacturing (BOM/Work Order/Quality), HR/Payroll, POS, Website/eCommerce/Portal**.
- No **Quotation, Delivery Note, Purchase Receipt, returns, or custom-field UI**.

For the precise gap between the 10 conceptual approval workflows and the 4 that are real maker/checker
engines, see **[docs/WORKFLOW_PARITY_REPORT.md](docs/WORKFLOW_PARITY_REPORT.md)**.

## Architecture at a glance

A **modular monolith** in five projects (four in the solution + a separate Playwright harness):

```
LafErp.Core      domain entities + enums (no dependencies)
LafErp.Data      ErpDbContext, EF Core mapping (42 tables), SqlServer + Sqlite
LafErp.Services  business logic (accounting, stock, workflow, rbac, ...)
LafErp.Web       ASP.NET Core MVC + minimal-API REST + Razor views (DI root)
LafErp.Tests     xUnit (74 tests)
playwright/      12 TypeScript browser smoke tests (separate from the .slnx)
```

Dependency direction is strictly inward: `Web → Services → Data → Core`. See
[docs/ARCHITECTURE.md](docs/ARCHITECTURE.md).

## Build, run, test

Prerequisites: **.NET 10 SDK**. No database server is required for the default run.

### Build

```bash
dotnet build LAF-EnterpriseERP.slnx -c Release
```

The full solution builds with **0 errors**.

### Run

```bash
dotnet run --project src/LafErp.Web
```

On startup the app **creates the schema (`EnsureCreated`) and seeds** a demo company (chart of
accounts, roles, workflows, demo masters) plus a handful of small demo transactions, so every page
shows live data on first launch.

- **Default storage: SQLite** — a local `laferp.db` file is created in the working directory. **No
  external services are needed.**
- **SQL Server:** set `ConnectionStrings:Default` (env var `ConnectionStrings__Default` or
  `appsettings`) and the app uses SQL Server instead. The model carries a `rowversion` concurrency
  token only under SQL Server.

**Ports:**

| Context | URL |
| --- | --- |
| `dotnet run` (Development profile, `launchSettings.json`) | `http://localhost:5027` (and `https://localhost:7032`) |
| Playwright harness (`--urls`) | `http://localhost:5080` |

> Schema dump: `dotnet run --project src/LafErp.Web -- schema` prints the SQL Server DDL for the
> model and exits without touching a database. The committed dump is in
> [`database/schema.sql`](database/schema.sql).

### Test

```bash
# .NET unit + integration tests (74)
dotnet test LAF-EnterpriseERP.slnx -c Release

# Browser smoke tests (12) — boots the real app on :5080
cd playwright && npm install && npx playwright test
```

See [tests/TEST_PLAN.md](tests/TEST_PLAN.md) and [tests/API_TEST_RESULTS.md](tests/API_TEST_RESULTS.md).
