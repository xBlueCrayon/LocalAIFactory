# Final Release Research Log

**Date:** 2026-06-21
**Author:** Principal DevOps / security engineer (final pre-release check)
**Internet access during this research:** YES. Web search was intermittently unavailable, so findings
were verified primarily by fetching the official documentation pages directly.

This is a **final check**, not an exhaustive survey. Findings are scoped to what actually affects
LocalAIFactory: a private, local-first .NET 10 / ASP.NET Core MVC + MSSQL/EF Core platform with
optional Ollama/Qdrant, Windows-auth RBAC, deployed via IIS or run locally on SQL Express / LocalDB.

### Official doc domains consulted (verified by fetch)

- `learn.microsoft.com` — ASP.NET Core on IIS (host/deploy + troubleshoot), EF Core applying migrations,
  SQL Server backup compression, Microsoft Entra ID / OIDC ASP.NET Core tutorial.
- `playwright.dev` — CI / headless guidance.
- `ollama.readthedocs.io` — Ollama FAQ (API, offline operation, OLLAMA_HOST).
- `qdrant.tech` — local installation (Docker, ports, health endpoints).
- `docs.docker.com` — Docker Desktop Windows install requirements.
- `csrc.nist.gov` / `nist.gov` — NIST SSDF (SP 800-218) and AI RMF 1.0.

## Net assessment

The repo's architecture already matches official guidance for the things that matter most in a
local-first banking-estate tool: optional external services are gated and health-cached off the
request path, secrets are out of the repo and encrypted at rest, RBAC is deny-by-default with an
append-only audit, and AI output is treated as a proposal. The two honest "watch items" are (1) the
startup `Database.Migrate()` path — acceptable for single-instance local installs and made safer by
EF Core 9+ migration locking, but a script/bundle is the production-grade alternative — and (2) the
IIS deploy path, where the **Hosting Bundle** and **app-pool SQL login** are the usual failure points.
