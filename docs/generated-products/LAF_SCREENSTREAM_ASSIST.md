## Publish / run / test (local LAN)

From the repository root:

```powershell
# Build the generated product (source-only in repo)
dotnet build generated-products/LAF-ScreenStreamAssist

# Publish the server to the local test folder (creates C:\LAFScreenStreamAssist\...)
pwsh generated-products/LAF-ScreenStreamAssist/scripts/publish-local-test-folder.ps1

# Start the server
C:\LAFScreenStreamAssist\Server\Start-Server.bat
# or run the EXE directly:
C:\LAFScreenStreamAssist\Server\LAFScreenStream.Server.exe
```

The client EXE is generated by the running server; the operator/viewer connects to the server over
the LAN.

## Production blockers (why LAN-only, not internet-ready)

- **No TLS/WSS** — traffic is not transport-encrypted, so it must stay on a trusted LAN.
- **No code-signing** — the EXEs are unsigned; SmartScreen/AV friction off-LAN.
- **No operator relay / port-forward** — no built-in NAT traversal or relay for internet use.

These cap production-grade at ~72%.

## Safety rules

- **Consent-based.** Screen-share starts only with explicit consent on the shared machine.
- **LAN-only.** Do not expose the server to the public internet until TLS/WSS, code-signing, and a
  relay/port-forward story exist.
- **Trusted network.** Because traffic is unencrypted, only run on a network you control.
